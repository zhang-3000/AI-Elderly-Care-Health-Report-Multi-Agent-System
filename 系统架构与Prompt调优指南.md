## 🏗️ 系统整体架构

### 核心设计理念

**多 Agent 协作流水线**：将复杂的健康评估任务分解为 7 个专业 Agent，每个 Agent 专注于特定任务，通过流水线方式协作完成。

```
用户数据输入
    ↓
┌─────────────────────────────────────────────────────────┐
│            OrchestratorAgentV2 (调度中心)                │
│                 协调 7 个 Agent 执行                      │
└─────────────────────────────────────────────────────────┘
    ↓
Stage 1: StatusAgent (状态判定)
    ↓
Stage 2: RiskAgentV2 (风险预测)
    ↓
Stage 3: FactorAgentV2 (因素分析)
    ↓
Stage 4: ActionPlanAgent (行动计划生成)
    ↓
Stage 5: PriorityAgent (优先级排序)
    ↓
Stage 6: ReviewAgent (反思校验)
    ↓
Stage 7: ReportAgentV2 (报告生成)
    ↓
最终报告输出
```

## 🔍 7个Agent详细介绍

### Stage 1: StatusAgent - 状态判定 Agent

**代码位置**: `multi_agent_system_v2.py` 第 139-210 行

#### 职责
根据 BADL（基本日常生活活动）和 IADL（工具性日常生活活动）数据，判定老人的失能状态。

#### 输入数据
- **健康限制** (e0): 过去6个月是否因健康问题限制了活动
- **BADL** (6项): 洗澡、穿衣、上厕所、室内活动、大小便控制、吃饭
- **IADL** (8项): 串门、购物、做饭、洗衣、步行1km、提重物、蹲起、公共交通

#### 输出结果
```json
{
    "status": 0/1/2,
    "status_name": "功能完好/需要部分协助/需要全面照护",
    "status_description": "生活自理/需要较多协助/需要全面照护支持",
    "health_limitation_impact": "健康限制对功能的影响说明",
    "badl_depend_count": 3,
    "iadl_unable_count": 5,
    "badl_details": ["洗澡", "穿衣"],
    "iadl_details": ["购物", "公共交通"],
    "explanation": "判定依据说明"
}
```

#### Prompt 调优要点
1. **判定标准的准确性**: 确保 BADL/IADL 的判定逻辑符合医学标准
2. **健康限制的权重**: health_limitation 如何影响最终判定
3. **解释的通俗性**: explanation 要让家属能看懂
4. **边界情况处理**: 如何处理数据缺失或模糊的情况

#### 调优方向
- [ ] 优化判定标准的描述清晰度
- [ ] 增强对边界情况的处理
- [ ] 改进 explanation 的语言风格

### Stage 2: RiskAgentV2 - 风险预测 Agent

**代码位置**: `multi_agent_system_v2.py` 第 214-304 行

#### 职责
预测老人的短期（1-4周）和中期（1-6月）健康风险，给出具体的触发场景和预防要点。

#### 输入数据
- 当前失能状态
- 基本信息（年龄、性别、居住地、教育）
- 健康限制
- 慢性病情况（7种）
- 认知功能（6项指标）
- 心理状态（抑郁、焦虑、孤独）
- 生活方式（吸烟、饮酒、锻炼、睡眠）
- 生理指标（体重、身高、视力、听力）
- 社会支持（居住、照护、医保）

#### 输出结果
```json
{
    "short_term_risks": [
        {
            "timeframe": "1-4周",
            "risk": "跌倒/骨折",
            "trigger": "夜间起床、卫生间、门槛",
            "severity": "高/中/低",
            "preventable": true/false,
            "prevention_key": "安装扶手、夜灯"
        }
    ],
    "medium_term_risks": [
        {
            "timeframe": "1-6月",
            "risk": "营养不足→肌少→更易跌倒",
            "chain": "营养↓ → 肌肉↓ → 行动↓ → 跌倒↑",
            "severity": "中",
            "preventable": true
        }
    ],
    "overall_risk_level": "低/中/高",
    "risk_summary": "风险总结"
}
```

#### Prompt 调优要点
1. **风险识别的全面性**: 是否覆盖了所有重要风险
2. **触发场景的具体性**: trigger 要具体到实际场景
3. **因果链的清晰性**: chain 要能让人理解风险演变过程
4. **可预防性判断**: 准确区分可预防和不可预防的风险
5. **优先级排序**: 如何判断风险的严重程度

#### 调优方向
- [ ] 补充更多常见风险类型
- [ ] 优化触发场景的描述
- [ ] 改进因果链的表达方式
- [ ] 增强风险严重度的判断逻辑

### Stage 3: FactorAgentV2 - 因素分析 Agent

**代码位置**: `multi_agent_system_v2.py` 第 308-422 行

#### 职责
生成"健康画像"，清晰呈现老人"好在哪里、短板在哪里"。

#### 输入数据

- StatusAgent 的判定结果
- RiskAgent 的风险评估结果

#### 特殊功能
- **自动计算 BMI**: 根据身高体重计算 BMI 并给出营养状态评估

#### 输出结果
```json
{
    "functional_status": {
        "level": "功能完好/需要部分协助/需要全面照护",
        "description": "可独立完成穿衣、进食、洗漱、如厕"

    "strengths": [

    ],
    "main_problems": [
        {
            "problem": "心脏病",
            "impact": "需要明确类型与严重程度",
            "priority": 1
        }
    ],
    "unchangeable_factors": ["年龄", "性别"],
    "changeable_factors": ["体力活动", "慢性病管理"]

```

#### Prompt 调优要点
1. **健康画像的全面性**: 是否准确反映了用户的整体健康状况
2. **优势识别**: 如何识别和强调保护因素
3. **问题优先级**: 如何判断哪些问题最重要
4. **可改变性判断**: 准确区分可改变和不可改变因素
5. **语言的积极性**: 既要指出问题，又要给予希望

#### 调优方向
- [ ] 优化优势识别的逻辑
- [ ] 改进问题优先级排序
- [ ] 增强对 BMI 等生理指标的解读
- [ ] 改善语言的平衡性（问题 vs 优势）

### Stage 4: ActionPlanAgent - 行动计划生成 Agent ✨新增

**代码位置**: `multi_agent_system_v2.py` 第 426-475 行

#### 职责
将健康建议转化为**可执行的行动计划**，每个行动都有负责人、具体步骤、完成标准。

#### 输入数据
- 用户画像
- 状态判定结果
- 风险预测结果
- 因素分析结果

#### 输出结果
```json
{
    "actions": [
        {
            "action_id": "A1",
            "title": "把"就医通道"先建立起来",
            "subtitle": "医保 + 就近固定就诊点",
            "responsible": "家属/主要照护者",
            "how_to_do": [
                "带身份证、户口本去村委会办理医保",
                "去乡镇卫生院建档"
            ],
            "completion_criteria": "明确以后看病去哪里、谁负责陪诊",
            "timeframe": "1-2周内完成",
            "difficulty": "中",
            "cost": "低",
            "impact": "高",
            "category": "医疗保障"
        }
    ]
}
```

#### Prompt 调优要点 ⭐ 重点
1. **可执行性**: 每个行动是否真的可以直接执行
2. **具体性**: how_to_do 是否足够具体（避免"建议锻炼"这种模糊表述）
3. **可验证性**: completion_criteria 是否可以验证
4. **负责人明确性**: 是否清楚谁来做
5. **情境适配性**: 是否考虑了城乡、独居/同住等实际情况
6. **行动数量**: 生成 8-12 个行动，覆盖 6 个类别

#### 调优方向 ⭐ 核心优化点
- [ ] 增强行动的可执行性（最重要）
- [ ] 优化 how_to_do 的具体程度
- [ ] 改进完成标准的可验证性
- [ ] 增加对不同情境的适配
- [ ] 补充更多行动类别和模板

### Stage 5: PriorityAgent - 优先级排序 Agent ✨新增

**代码位置**: `multi_agent_system_v2.py` 第 479-527 行

#### 职责
基于风险紧急度、可行性、成本效益，将行动计划智能排序为 A/B/C 三级。

#### 排序逻辑
```python
优先级得分 = (
    风险严重度 × 0.4 +
    可预防性 × 0.3 +
    可行性 × 0.2 +
    成本效益比 × 0.1
)
```

#### 输出结果
```json
{
    "priority_a": [
        {
            "action_id": "A3",
            "rank": 1,
            "reason": "成本低但收益最大，立即可做",
            "urgency": "立刻"
        }
    ],
    "priority_b": [...],
    "priority_c": [...],
    "排序说明": "简要说明排序逻辑"
}
```

#### 分级原则
- **A级（第一优先）**: 3项以内，"最要命、最可防、最好做"
- **B级（第二优先）**: 4-7项，日常维持类
- **C级（第三优先）**: 长期改善类

#### Prompt 调优要点
1. **排序合理性**: 排序是否符合实际需求
2. **理由清晰性**: reason 是否能让人信服
3. **数量控制**: A/B/C 级的数量是否合理
4. **紧急度判断**: urgency 的判断是否准确
5. **权重平衡**: 四个维度的权重是否合理

#### 调优方向
- [ ] 优化排序算法的权重
- [ ] 改进排序理由的表达
- [ ] 调整各级的数量分配
- [ ] 增强对特殊情况的处理

### Stage 6: ReviewAgent - 反思校验 Agent

**代码位置**: `multi_agent_system_v2.py` 第 531-585 行

#### 职责
检查报告的准确性、一致性和安全性，识别需要紧急就医的情况。

#### 检查要点
1. **一致性**: 状态判定与风险预测是否矛盾
2. **合理性**: 行动计划是否与风险因素匹配
3. **安全性**: 是否有需要紧急就医的情况
4. **可执行性**: 行动计划是否具体可落地
5. **完整性**: 是否覆盖了所有必要内容

#### 紧急情况识别
- 突发胸痛、呼吸困难
- 新发偏瘫或言语不清
- 急性意识改变
- 严重抑郁、自伤想法
- 快速功能下降（几周内）

#### 输出结果
```json
{
    "consistency_check": {"passed": true, "issues": []},
    "safety_check": {"urgent": false, "urgent_reason": ""},
    "executability_check": {"passed": true, "issues": []},
    "completeness_check": {"passed": true, "missing": []},
    "suggestions": ["改进建议"],
    "overall_quality": "优/良/中/差",
    "approved": true
}
```

#### Prompt 调优要点
1. **检查的全面性**: 是否覆盖了所有重要检查点
2. **问题识别的准确性**: 能否准确发现问题
3. **建议的可操作性**: suggestions 是否有用
4. **紧急情况判断**: 安全性检查是否到位

#### 调优方向
- [ ] 补充更多检查维度
- [ ] 优化问题识别逻辑
- [ ] 改进建议的质量
- [ ] 增强紧急情况识别

### Stage 7: ReportAgentV2 - 报告生成 Agent

**代码位置**: `multi_agent_system_v2.py` 第 589-705 行

#### 职责
将所有评估结果整合为最终的"健康评估与照护行动计划"报告。

#### 报告结构
```markdown
# 健康评估与照护行动计划

## 0. 报告说明
免责声明和使用说明

## 1. 健康报告总结（3句话）
- 整体功能评估
- 核心问题
- 干预建议概要

## 2. 您的健康画像
### （1）功能状态
### （2）优势（需要继续保持）
### （3）主要问题（本报告重点要解决的）

## 3. 风险因素
### 近期（1-4周）重点风险
### 中期（1-6月）重点风险

## 4. 健康建议
### A. 第一优先级
### B. 第二优先级
### C. 第三优先级

## 5. 温馨寄语
```

#### 语言风格要求
- **口语化、接地气**
- **避免医学术语**
- **强调"可以做什么"**
- **每个建议都有负责人、怎么做、完成标准**

#### Prompt 调优要点 ⭐ 重点
1. **3句话总结**: 是否简洁有力
2. **语言风格**: 是否口语化、易懂
3. **结构清晰性**: 是否层次分明
4. **温馨寄语**: 是否温暖、鼓励
5. **整体连贯性**: 各部分是否衔接自然

#### 调优方向 ⭐ 核心优化点
- [ ] 优化语言风格（更口语化）
- [ ] 改进3句话总结的质量
- [ ] 增强温馨寄语的感染力
- [ ] 调整报告的整体结构

## 🔧 调优技巧

### 1. Prompt 结构建议
```
你是[角色定位]。[核心任务描述]。

【输入数据说明】
- 数据1: 说明
- 数据2: 说明

【输出格式】JSON
{具体格式示例}

【核心原则】
1. 原则1
2. 原则2

【注意事项】
- 注意1
- 注意2
```

### 2. 常见问题处理

**问题1: 输出格式不稳定**
- 解决: 在 Prompt 中明确要求 JSON 格式，给出完整示例

**问题2: 语言风格不符合预期**
- 解决: 在 Prompt 中明确语言风格要求（口语化、专业、温和等）

**问题3: 输出内容不够具体**
- 解决: 在 Prompt 中给出具体示例，强调"具体"、"可执行"

**问题4: 遗漏重要信息**
- 解决: 在 Prompt 中列出必须包含的要素清单

### 3. 测试数据
- **状态0样本**: 81岁男性（无失能）
- **状态1样本**: 100岁男性（部分失能）
- **状态2样本**: 102岁女性（严重失能）
