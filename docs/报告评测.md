# 报告评测系统

## 一、常规指标体系概述

| 类型                   | 指标                                                                                                     | 参考文献                                                                                                    | 要求     |
| ---------------------- | -------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------- | -------- |
| 文本质量（相似度指标） | ROUGE、BLEU、BERTScore、BLEURT                                                                           | https://aclanthology.org/2023.clinicalnlp-1.48/ ; https://openreview.net/forum?id=SkeHuCVFDr                | 需标注数据 |
| 医学/事实正确性        | Medical Concept Coverage、Negation Correctness、Fact Precision/Recall、Hallucination Rate、Omission Rate | https://aclanthology.org/2023.clinicalnlp-1.48/                                                             | 需标注数据 |
| 检索增强生成（RAG）相关指标 | Faithfulness、Answer Relevance、Context Relevance                                                   | https://arxiv.org/abs/2309.15217                                                                            | **不需** |

我们的重点是第三类——不需要专家标注参考报告即可自动化运行的 RAG 相关指标。

---

## 二、理论基础

来源：**RAGAS: Automated Evaluation of Retrieval Augmented Generation**（https://arxiv.org/abs/2309.15217 ）

该论文提出了三个完全不需要专家标注的评测指标。我们在实现中对 Answer Relevance 进行了适配（见下文），其余两个指标忠实遵循原始论文设计。

### 2.1 Faithfulness（忠实度）

**评估对象**：生成模块——报告中是否存在编造内容（幻觉检测）。

**原理**：

1. 用 LLM 把报告拆解为若干条独立的事实陈述（如"该老人患有高血压"、"建议加强跌倒防护"）。
2. 逐条对照 RAG 检索到的原始上下文，由 LLM 判断该陈述是否有上下文依据。
3. 得分 = 被上下文支持的陈述数 ÷ 总陈述数，范围 0–1。

**性能**：WikiEval 数据集上与人工判断一致率 **0.95**（优于 GPT 直接打分的 0.72）。

### 2.2 Answer Relevance → Profile Coverage（画像覆盖度）

**原始设计**：RAGAS 的 Answer Relevance 采用"逆向生成验证"——从报告反推假设问题，与原始问题做嵌入相似度。

**我们的适配**：本系统的输入不是一个问题，而是结构化的老人画像（UserProfile）。直接套用反向生成方式，需要先从画像构造一个人工 query，引入额外的不确定性。因此我们将该指标改为 **Profile Coverage（画像覆盖度）**——直接检查报告是否覆盖了画像中的每个关键要素。

**评估对象**：生成模块——报告是否完整回应了老人的实际情况，有没有遗漏关键信息。

**原理**：

1. 从 UserProfile 自动提取关键要素清单（慢性病、功能受限项、心理状态等）。
2. 由 LLM 逐条判断报告中是否提及或覆盖了该要素。
3. 得分 = 被覆盖的要素数 ÷ 总要素数，范围 0–1。

### 2.3 Context Relevance（上下文相关性）

**评估对象**：检索模块——RAG 检索出来的知识是否精准，有没有太多冗余无关内容。

**原理**（参考 *Lost in the Middle*, https://arxiv.org/abs/2307.03172 ）：

1. 将检索到的上下文按句分割。
2. 由 LLM 从上下文中摘出对回答查询真正有用的句子。
3. 得分 = 有用句子数 ÷ 总句子数，范围 0–1。

**性能**：WikiEval 上与人工判断一致率 **0.70**（三个指标中最难自动化的一个）。

### 指标总结

| 指标               | 评估环节 | 评估内容                                         | 是否需要 RAG 上下文 | LLM 调用次数 |
| ------------------ | -------- | ------------------------------------------------ | -------------------- | ------------ |
| Faithfulness       | 生成模块 | AI 是否在报告中编造了不存在的信息                | 是                   | 2            |
| Profile Coverage   | 生成模块 | 报告是否覆盖了老人画像中的关键健康要素           | 否                   | 1            |
| Context Relevance  | 检索模块 | RAG 检索出来的知识是否精准有效，有没有冗余       | 是                   | 1            |

## 三、实现说明

### 3.1 代码结构

```
backend/code/
├── evaluation/                  # 评测模块
│   ├── __init__.py
│   ├── utils.py                 # 共享工具（LLM 调用、分句、画像要素提取）
│   ├── metrics.py               # 三个指标的具体实现
│   └── evaluator.py             # 评测编排器 ReportEvaluator
├── evaluate_report.py           # CLI 评测入口脚本
└── rag/
    └── agent.py                 # 已修改：retrieve() 返回完整 text 字段
```

### 3.2 RAG 检索增强

原来 `PageIndexRAGAgent.retrieve()` 返回的每条 hit 只包含 `excerpt`（约 280 字符的摘要），不包含 chunk 的完整文本。评测需要完整的检索内容作为上下文验证依据。

**修改内容**（`rag/agent.py`）：在 `retrieve()` 返回的 hit 字典中新增 `text` 字段，直接取自索引中 chunk 的完整文本（通常 1000–3000 字符）。原有的 `excerpt` 字段和 LLM prompt 构建逻辑不受影响。

### 3.3 各指标实现细节

#### Faithfulness（`FaithfulnessMetric`）

```
输入: report_text (报告全文), retrieved_context (检索上下文)
输出: FaithfulnessResult { score, total_statements, supported_statements, statements[] }
```

处理流程：

1. **陈述提取**：调用 LLM，将报告拆解为独立的事实陈述。过滤修辞性/鼓励性语句，只保留可判断真假的事实句（长度 > 8 字符）。
2. **批量验证**：将所有陈述编号后与检索上下文一起提交给 LLM，要求对每条陈述输出 `{"index", "supported", "reason"}` 的 JSON 数组。内置回退解析逻辑，当 LLM 未返回有效 JSON 时按行判断。
3. **计分**：`score = supported_count / total_count`。

#### Profile Coverage（`ProfileCoverageMetric`）

```
输入: report_text (报告全文), profile (用户画像字典)
输出: ProfileCoverageResult { score, total_elements, covered_elements, elements[] }
```

处理流程：

1. **要素提取**（`extract_profile_elements()`）：从画像字典中自动提取关键要素，包括：
   - 基本信息：年龄、性别
   - 慢性病：高血压、糖尿病、心脏病、中风、关节炎、肿瘤
   - BADL 受限项：洗澡、穿衣、上厕所、室内移动、大小便控制、吃饭
   - IADL 受限项：串门、买东西、做饭、洗衣服、走远路、提重物、蹲起、乘坐交通
   - 认知状态、心理状态（抑郁、孤独）、感官（视力、听力）、生活习惯（吸烟、锻炼）
2. **覆盖判断**：将要素清单与报告全文提交给 LLM，逐项判断报告是否"直接提及或间接涉及"该要素，并给出报告中的对应证据片段。
3. **计分**：`score = covered_count / total_count`。

#### Context Relevance（`ContextRelevanceMetric`）

```
输入: query_description (查询描述), retrieved_context (检索上下文)
输出: ContextRelevanceResult { score, total_sentences, useful_sentences, useful_sentence_list[] }
```

处理流程：

1. **分句**：按中文句号、叹号、问号和换行符分割检索上下文，过滤长度 ≤ 5 的片段。
2. **有用句提取**：LLM 从上下文中摘出对回答查询有用的原文句子。如果没有有用内容，返回"信息不足"。
3. **计分**：`score = useful_count / total_count`，上限为 1.0。

### 3.4 评测编排器（`ReportEvaluator`）

`ReportEvaluator` 提供两种评测模式：

| 模式         | 方法                     | 说明                                               |
| ------------ | ------------------------ | -------------------------------------------------- |
| **在线评测** | `evaluate(results, profile)` | 在报告生成流程中直接调用，`results` 来自 orchestrator |
| **离线评测** | `evaluate_from_file(json_path)` | 从保存的 JSON 报告文件加载后评测                    |

离线评测时，如果原始报告没有 RAG 数据（`knowledge` 为空），可通过 `re_retrieve=True` 参数让评测器重新运行 RAG 检索——从 JSON 中恢复 `UserProfile`，调用 `KnowledgeAgent.retrieve_comprehensive()` 获取上下文。

三个指标可独立开关（`run_faithfulness`、`run_coverage`、`run_context_relevance`），Faithfulness 和 Context Relevance 在无 RAG 上下文时自动跳过。

评测结果通过 `EvaluationResult` 返回：

```python
result.summary()    # {"faithfulness": 0.82, "profile_coverage": 0.91, "context_relevance": 0.65}
result.to_dict()    # 完整结果，含每条陈述/要素的详细判定
```

## 四、使用方式

### 4.1 CLI 命令行

```bash
cd backend/code

# 评测单个报告（仅 Profile Coverage，不依赖 RAG）
python evaluate_report.py -i ../data/reports/202603/report_xxx.json -m coverage

# 评测单个报告（全部指标，重新运行 RAG 检索）
python evaluate_report.py -i ../data/reports/202603/report_xxx.json --re-retrieve

# 评测整个目录
python evaluate_report.py -i ../data/reports/202603/

# 指定输出路径
python evaluate_report.py -i report.json -o eval_result.json

# 显示详细日志
python evaluate_report.py -i report.json --re-retrieve -v
```

CLI 参数说明：

| 参数              | 说明                                                     |
| ----------------- | -------------------------------------------------------- |
| `--input, -i`     | 报告 JSON 文件路径，或包含报告的目录                     |
| `--output, -o`    | 评测结果输出路径（默认在输入文件同目录生成 `eval_*.json`） |
| `--re-retrieve`   | 重新运行 RAG 检索（原始报告无 RAG 数据时需要）           |
| `--metrics, -m`   | 要运行的指标，逗号分隔：`faithfulness,coverage,context_relevance` |
| `--verbose, -v`   | 显示详细日志                                             |

### 4.2 Python 代码调用

```python
from evaluation.evaluator import ReportEvaluator

evaluator = ReportEvaluator()

# 方式一：在线评测（在报告生成流程中）
result = evaluator.evaluate(results, profile_dict)
print(result.summary())

# 方式二：离线评测（从 JSON 文件）
result = evaluator.evaluate_from_file("report.json", re_retrieve=True)
print(result.to_dict())
```

### 4.3 环境变量

评测模块复用主系统的 DeepSeek 配置，需要设置以下环境变量：

| 变量                       | 说明                        | 默认值                    |
| -------------------------- | --------------------------- | ------------------------- |
| `DEEPSEEK_API_KEY`         | DeepSeek API 密钥           | （必填）                  |
| `DEEPSEEK_BASE_URL`        | DeepSeek API 地址           | `https://api.deepseek.com` |
| `DEEPSEEK_MODEL`           | 模型名称                    | `deepseek-chat`           |
| `DEEPSEEK_TIMEOUT_SECONDS` | 请求超时（秒）              | `180`                     |
| `RAG_INDEX_PATH`           | RAG 索引文件路径（重新检索时用） | `data/rag_indexes/combined_index.json` |

## 六、LLM 调用成本

每份报告评测共约 **4 次** DeepSeek API 调用：

| 指标              | 调用次数 | 说明                       |
| ----------------- | -------- | -------------------------- |
| Faithfulness      | 2        | 1 次陈述提取 + 1 次批量验证 |
| Profile Coverage  | 1        | 1 次批量覆盖判断           |
| Context Relevance | 1        | 1 次有用句提取             |

按 DeepSeek 当前定价，单份报告评测成本极低（不到 0.01 元）。

## 七、指标解读与注意事项

### 分数参考范围

| 指标              | 优秀    | 良好      | 需关注   |
| ----------------- | ------- | --------- | -------- |
| Faithfulness      | > 0.85  | 0.70–0.85 | < 0.70   |
| Profile Coverage  | > 0.90  | 0.75–0.90 | < 0.75   |
| Context Relevance | > 0.70  | 0.50–0.70 | < 0.50   |

> 以上范围为经验参考值，具体阈值需要根据实际评测数据积累后校准。

### 注意事项

1. **Faithfulness 和 Context Relevance 依赖 RAG 上下文**。如果报告生成时未启用 RAG（`RAG_ENABLED=false`），这两个指标需要通过 `--re-retrieve` 重新检索才能评测。
2. **Profile Coverage 不依赖 RAG**，可以独立评测任何已生成的报告。
3. **Faithfulness 的局限**：该指标只衡量"报告内容是否有据可查"，不衡量内容的医学正确性。如果知识库本身有错误，Faithfulness 分数仍然可能很高。
4. **Context Relevance 是对检索模块的评估**，分数低说明检索不够精准（返回了大量无关内容），需要优化检索策略或知识库索引，而不是调整报告生成逻辑。
5. 评测结果受 LLM 判断能力影响，建议对同一份报告多次评测取平均值以提高稳定性。
